# Banzami Platform — Branding & UI Consistency Audit

**Date:** 2026-05-15  
**Scope:** Full platform — Flutter mobile, Flutter SDK, TypeScript SDK, web apps (dashboard, admin, pay), WooCommerce plugin, documentation  
**Auditor:** Engineering review

---

## 1. Executive Summary

The Banzami platform has **two active color systems in conflict**, a **missing secondary and accent palette** across all implementations, and **hardcoded color values** on production-facing payment pages. The Flutter SDK and web apps diverge from each other and both diverge from the official design system specification on critical tokens.

**Official Design System (source of truth for this audit):**

| Token | Name | Value |
|---|---|---|
| Primary | Space Cherry | `#990011` |
| Secondary | Wine Rose | `#A63A50` |
| Accent | Savanna Gold | `#C89B3C` |
| Background | — | `#FCF6F5` |
| Surface | — | `#FFFFFF` |
| Text Primary | — | `#1A1A1A` |
| Text Secondary | — | `#6B6B6B` |

**Verdict:** The Flutter SDK primary (`#990011`) is correct. The TypeScript SDK primary (`#6D071A`) is wrong. The secondary color (Wine Rose `#A63A50`) does not exist in any SDK. The accent color (Savanna Gold `#C89B3C`) does not exist in any SDK. Text Primary is wrong in Flutter SDK (`#1C0D0D` instead of `#1A1A1A`).

---

## 2. Strengths

- Flutter SDK has a **solid, systematic design token architecture** — BanzaColors, BanzaTypography, BanzaSpacing, BanzaRadius, BanzaShadow are all defined and used consistently in mobile screens.
- Mobile app screens (consumer and merchant) have **zero hardcoded colors** — all reference BanzaColors constants.
- Spacing and typography systems are **consistent and correct** across all platforms.
- The Flutter SDK ThemeData is well-configured: AppBar, buttons, inputs, cards, and navigation bar all use theme constants.
- The merchant and consumer app navigation structure is clean and follows the design language.
- The `switch_firebase_config.sh` build script correctly selects the Firebase plist per flavor without hardcoded logic in app code.

---

## 3. Weaknesses

- The TypeScript SDK was built with a different primary color (`#6D071A`) that made it into the brand documentation, overriding the correct Flutter value (`#990011`).
- Neither SDK defines the **secondary color (Wine Rose `#A63A50`)** or the **accent color (Savanna Gold `#C89B3C`)**.
- Text Primary is `#1C0D0D` in Flutter SDK instead of the correct `#1A1A1A`.
- Text Secondary (`#6B6B6B`) is not a named constant anywhere.
- The pay app (`apps/pay`) hardcodes the wrong primary (`#6D071A`) on production payment pages.
- The WooCommerce plugin uses the wrong primary color (`#6D071A`) on customer-facing checkout pages.
- The brand documentation (`docs/brand/README.md`) now reflects the wrong color set.

---

## 4. Critical UI Inconsistencies

### C-01 — TypeScript SDK primary color is wrong

**Files:**  
- `sdk/typescript/src/theme/index.ts`

**Current:** `wine: '#6D071A'`  
**Correct:** `wine: '#990011'`  

**Impact:** Any checkout component rendered by the TypeScript SDK (payment links, embedded checkout) shows a noticeably darker, more aggressive burgundy than the mobile app and dashboard. This is directly visible to paying customers.

---

### C-02 — Pay app hardcodes wrong primary on production payment pages

**Files:**  
- `apps/pay/app/[slug]/pay-client.tsx` — 4 occurrences (lines ~126, ~188, ~212, ~258)  
- `apps/pay/app/page.tsx` — 1 occurrence (line ~7)

**Current:** `style={{ background: '#6D071A' }}`  
**Correct:** Use Tailwind class `bg-wine` (which resolves to `#990011` via the pay app's `tailwind.config.ts`)

**Impact:** The pay page — the screen a customer sees when making a payment — displays a different primary than the mobile app and dashboard simultaneously. This is the most visible branding failure on the critical payment path.

---

### C-03 — WooCommerce plugin uses wrong primary on e-commerce checkout

**Files:**  
- `plugins/woocommerce/banzami-payment/templates/payment-page.php` — lines ~24 (header bg), ~43 (button bg), ~80 (QR color)

**Current:** `#6D071A`  
**Correct:** `#990011`

**Impact:** A customer paying through WooCommerce sees one primary color on the merchant's site and a different one when they open the Banzami mobile app. Destroys visual trust at the moment of payment.

## 5. Branding Inconsistencies

### B-01 — Secondary color (Wine Rose) missing from all implementations

**Affected:** All SDKs, all apps, all plugins

The official secondary color **Wine Rose `#A63A50`** is not defined anywhere. The Flutter SDK has `wineLight: '#B5001A'` which is a lighter variant, but not the secondary. The TypeScript SDK has `wineMedium: '#8E1026'` — also not the secondary.

**Fix:** Add to Flutter SDK (`banzami_theme.dart`):
```dart
static const wineRose = Color(0xFFA63A50); // Secondary
```
Add to TypeScript SDK (`theme/index.ts`):
```typescript
wineRose: '#A63A50',
```
Add to all Tailwind configs:
```typescript
wineRose: '#A63A50',
```

---

### B-02 — Accent color (Savanna Gold) missing from all implementations

**Affected:** All SDKs, all apps, all plugins

**Savanna Gold `#C89B3C`** is not defined anywhere. The TypeScript SDK defines `copper: '#C56A2D'` and `copperLight: '#D4834A'` — a different color family (more orange-red vs gold). The brand documentation (`docs/brand/README.md`) documents Copper, but the official design system now designates Savanna Gold as the accent.

**Fix:** Add to Flutter SDK:
```dart
static const savannaGold  = Color(0xFFC89B3C); // Accent
static const savannaGoldLight = Color(0xFFD4AF5C); // Hover/light variant
```
Add to all Tailwind configs:
```typescript
gold: {
  DEFAULT: '#C89B3C',
  light:   '#D4AF5C',
}
```
Remove or deprecate Copper from TypeScript SDK.

---

### B-03 — Text Primary wrong in Flutter SDK

**Files:**  
- `sdk/flutter/lib/theme/banzami_theme.dart`

**Current:** `gray900 = Color(0xFF1C0D0D)` — used as text primary  
**Correct:** `#1A1A1A`

**Current:** Text Secondary not defined as a named constant.  
**Correct:** Add `textSecondary = Color(0xFF6B6B6B)`

**Why it matters:** The current text color `#1C0D0D` has a red tint, giving text a slightly warm hue. The correct `#1A1A1A` is neutral near-black — cleaner for readability and more trustworthy on financial screens.

---

### B-04 — TypeScript SDK grays and off-white all differ from Flutter SDK

**Files:**  
- `sdk/typescript/src/theme/index.ts`

All neutral tokens diverge from the Flutter SDK. Since the official background is `#FCF6F5` (which is the Flutter SDK value), the TypeScript SDK's `offWhite: '#F6F4F1'` is wrong.

| Token | TypeScript (wrong) | Official |
|---|---|---|
| offWhite | `#F6F4F1` | `#FCF6F5` |
| gray100 | `#F0EDEA` | `#F5EEED` (Flutter value) |
| gray400 | `#9E9A96` | `#9C8483` (Flutter value) |
| gray700 | `#4A4744` | `#534040` (Flutter value) |
| gray900 | `#1A1816` | `#1A1A1A` (official spec) |

---

### B-05 — Error color inconsistency between platforms

| Platform | Error Color |
|---|---|
| Flutter SDK | `#DC2626` |
| TypeScript SDK | `#B91C1C` |
| Tailwind apps | `#DC2626` |

No standard has been declared. The difference is subtle but violates consistency. Pick one and apply it everywhere.

---

### B-06 — Brand documentation out of sync

**File:** `docs/brand/README.md`

The brand doc specifies `#6D071A` as Wine Primary (incorrect per official spec `#990011`), documents the Copper palette (which should be replaced by Savanna Gold), and does not mention Wine Rose as the secondary.

## 6. SDK Issues

### SDK-01 — Flutter SDK checkout screen has hardcoded semantic colors

**File:** `sdk/flutter/lib/screens/checkout_screen.dart`

| Line | Current | Correct |
|---|---|---|
| ~255 | `Color(0xFFECFDF5)` (success bg) | `BanzaColors.successBg` |

| ~286 | `Color(0xFFFEF2F2)` (error bg) | `BanzaColors.errorBg` |
| ~288 | `Colors.red` (error icon) | `BanzaColors.error` |

These constants are already defined in `banzami_theme.dart` but not referenced from `checkout_screen.dart`.

---

### SDK-02 — TypeScript SDK has Copper palette that conflicts with official accent

**File:** `sdk/typescript/src/theme/index.ts`

Copper (`#C56A2D`, `#D4834A`) should be replaced by Savanna Gold (`#C89B3C`, `#D4AF5C`). Copper is not referenced in any component code — it can be safely renamed without any runtime impact.

---

### SDK-03 — TypeScript SDK has no checkout UI component for QR codes

**File:** `sdk/typescript/src/`

The Flutter SDK exposes a `BanzamiQrWidget` with full theming. The TypeScript SDK has no equivalent QR component. Merchants integrating via web have no reusable, branded QR code component.

---

### SDK-04 — Flutter SDK missing Wine Rose and Savanna Gold makes accent usage impossible

Any widget that wants to use the secondary or accent color in the mobile app currently has no constant to reference. Developers are forced to hardcode or approximate.

## 7. UX Issues

### UX-01 — Savanna Gold never used as positive financial emphasis

The accent color (`#C89B3C`) was designed for: QR payment actions, completed payment states, positive financial emphasis. Currently all positive states use green (`#166534`). A warm gold accent on completed payments would reinforce the "African premium fintech" feel and differentiate Banzami visually.

---

### UX-02 — Consumer app lacks APNs permission request on first launch

**File:** `apps/mobile/lib/screens/main_screen.dart`

The consumer app requests push notification permission after login (inside `_startNotifications`). There is no onboarding screen that explains WHY the permission is needed before requesting it. iOS best practice is to show a custom explanation screen first, then trigger the native dialog. Without context, users are more likely to deny the permission.

---

### UX-03 — Consumer and merchant "empty state" patterns not audited for consistency

No shared `BanzamiEmptyState` widget exists in the SDK. Each screen implements its own empty state inline. Inconsistency is likely as the app grows.

---

## 8. Architecture / Design System Issues

### A-01 — No single source of truth for design tokens across platforms

Currently:
- Flutter tokens → `sdk/flutter/lib/theme/banzami_theme.dart`
- TypeScript tokens → `sdk/typescript/src/theme/index.ts`
- Web tokens → each app's `tailwind.config.ts`

These three are **manually kept in sync** (and currently are not in sync). A design token file (JSON or YAML) that generates all three outputs via a build step would eliminate this class of problem permanently.

**Recommended approach:** Use [Style Dictionary](https://amzn.github.io/style-dictionary/) or a simple `tokens.json` with a codegen script. All three platforms consume the same token definitions.

---

### A-02 — Tailwind config duplicated across three apps

**Files:**  
- `apps/dashboard/tailwind.config.ts`  
- `apps/admin/tailwind.config.ts`  
- `apps/pay/tailwind.config.ts`

All three define the same color palette manually. If the primary color changes, all three files must be updated separately — and the pay app already proved this fails (it has hardcoded wrong values).

**Fix:** Extract a shared `@banzami/tailwind-preset` package that all apps extend:
```typescript
// apps/*/tailwind.config.ts
import { banzamiPreset } from '@banzami/tailwind-preset';
export default { presets: [banzamiPreset], content: [...] };
```

---

### A-03 — No shared React component library for web

Dashboard and admin share no components. Both implement their own buttons, cards, tables, badges, etc. Any branding change must be applied to both apps separately.
