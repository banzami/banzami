# Financial Tracing

Version: 1.0 · Status: Stable · Domain: Observability

Every operation in Banza emits a causal chain of identifiers that links the
full lifecycle of a payment flow — from the triggering operation, through all
derived ledger postings and events, to eventual settlement.

---

## Identifiers

### trace_id

One UUID per top-level user-visible operation. It is created once and carried
unchanged through every derived operation in the same payment flow.

| Operation | When trace_id is created |
|-----------|--------------------------|
| Direct transfer (`POST /transfers`) | At transfer creation |
| Payment request (`POST /payment-requests`) | At request creation |
| QR code (`POST /qr`) | At QR generation |
| Settlement batch (`POST /settlement/batches`) | At batch creation |
| Wallet creation (`POST /wallets`) | At wallet creation |

When a payment request or QR code is paid, the resulting transfer **inherits**
the same `trace_id` as the original request. This means a single `trace_id`
covers the full payment lifecycle from intent to money movement.

### correlation_id

Identifies the current aggregate step within a trace. It changes as the
operation moves through layers.

- For a direct transfer: `correlation_id === trace_id`
- For ledger entries generated by a transfer: `correlation_id === transfer_id`
- For events emitted by a transfer: `correlation_id === transfer_id`

### causation_id

The ID of the operation that directly caused the current entity.

- A transfer caused by paying a payment request: `causation_id = pr_id`
- A transfer caused by scanning a QR code: `causation_id = qr_id`
- Top-level operations: `causation_id` is absent

## Causal chain — payment request example

```
POST /payment-requests   →  pr-abc   (trace_id: tr-xyz)
POST /payment-requests/pr-abc/pay

  creates transfer:
    transfer_id:    txfr-def
    trace_id:       tr-xyz          ← inherited from PR
    correlation_id: tr-xyz          ← correlation_id of the PR at creation
    causation_id:   pr-abc          ← the PR caused this transfer

  creates ledger entries (DEBIT + CREDIT):
    trace_id:       tr-xyz
    correlation_id: txfr-def        ← transfer is the ledger's correlation

  emits events (payment.sent, payment.received):
    trace_id:       tr-xyz
    correlation_id: txfr-def

GET /traces/tr-xyz →  timeline + all of the above, linked by trace_id
```

## Causal chain — QR payment example

```
POST /qr  →  qr-uvw   (trace_id: tr-qrs)
POST /qr/qr-uvw/pay

    causation_id:   qr-uvw          ← the QR caused this transfer

```

## API

### GET /traces

Returns all distinct trace IDs currently in memory.

```json
{
  "count": 3,
  "trace_ids": ["tr-abc...", "tr-def...", "tr-ghi..."]
}
```

### GET /traces/{trace_id}

Returns the full causal chain view for a trace.

```json
{
  "trace_id": "tr-xyz...",
  "timeline": [
    {
      "timestamp": "2026-05-28T10:00:00Z",
      "operation_type": "payment_request.created",
      "id": "pr-abc...",
      "summary": "Payment request created — 25000 → sandbox-merchant-1"
    },
    {
      "timestamp": "2026-05-28T10:01:00Z",
      "operation_type": "transfer.executed",
      "id": "txfr-def...",
      "summary": "sandbox-consumer-1 → sandbox-merchant-1 : 25000 minor units AOA"
    }
  ],
  "transfers": [ ... ],
  "ledger_entries": [ ... ],
  "events": [ ... ],
  "settlement_batches": [ ... ],
  "payment_requests": [ ... ],
  "qr_codes": [ ... ]
}
```

The `timeline` array is sorted by timestamp and includes every operation in the
causal chain: PR creation, QR generation, transfer execution, ledger postings,
event emission, settlement.

Returns `404` if no entity with that `trace_id` exists.

## Fields on each entity

Every stateful entity in the sandbox carries trace fields:

| Entity | Fields |
|--------|--------|
| `SandboxTransfer` | `trace_id`, `correlation_id`, `causation_id` (optional) |
| `SandboxPaymentRequest` | `trace_id` |
| `SandboxQrCode` | `trace_id` |
| `SandboxLedgerEntry` | `trace_id`, `correlation_id` |
| `SandboxSettlementBatch` | `trace_id` |
| `SandboxEvent` | `trace_id`, `correlation_id`, `causation_id` (optional), `aggregate_type`, `aggregate_id` |

## In the demo wallet

Every transfer row, payment request row, settlement batch row, and event row
shows a clickable `trace_id` chip. Clicking it opens the **Trace** tab with the
full causal chain pre-loaded.

The Trace tab also shows the full timeline view, grouped by transfers, ledger
entries, events, QR codes, payment requests, and settlement batches.

---

## In production

In a production Banza operator, these same identifiers appear in:

- Webhook payloads delivered to merchant backends
- Structured logs (include `trace_id` in every log line)
- Distributed tracing (map `trace_id` → OpenTelemetry trace)
- Reconciliation reports (group ledger entries by `trace_id`)
- Audit trails (full causal reconstruction from `trace_id`)

The sandbox makes the entire causal chain inspectable via HTTP so you can
understand the model before wiring it into observability infrastructure.
