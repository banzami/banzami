## Decision

The naming inversion is total. No protocol identifier is permanently protected.

The following wire format identifiers must be migrated from legacy (Banzami-protocol) to canonical (Banza-protocol) names. Domain names (`banzami.org`) and email addresses (`@banzami.org`) remain temporarily protected; everything else migrates.

---

## What Changes

### 1. Operator Manifest Discovery URL

| | Value |
|-|-------|
| **Legacy (deprecated)** | `/.well-known/banzami/operator.json` |
| **Canonical (new)** | `/.well-known/banza/operator.json` |

This is the URL path served by every certified operator to publish their operator manifest. All operator SDKs, validation tools, and discovery clients must update to the new path.

### 2. QR Payment Payload Prefixes

| | Production | Sandbox |
|-|-----------|---------|
| **Legacy (deprecated)** | `BANZAMI:{base64}` | `BANZAMI-SBX:{base64}` |
| **Canonical (new)** | `BANZA:{base64}` | `BANZA-SBX:{base64}` |

These prefixes appear in every QR code payload generated by the protocol. QR code generators must emit the new prefix. QR code validators must initially accept both prefixes (compatibility window), then deprecate the legacy prefix.

## Why It Is Breaking

These are **wire-level identifiers** â€” they appear in network traffic, stored payloads, and deployed configurations:

1. **QR codes are physical artifacts.** Every QR code ever printed or displayed uses `BANZAMI:` or `BANZAMI-SBX:`. Those codes cannot be retroactively changed. A scanner that only recognizes `BANZA:` will fail on legacy codes.

2. **Operator manifests are deployed URLs.** Every operator that has gone live has implemented `/.well-known/banzami/operator.json`. Any discovery client updated to fetch `/.well-known/banza/operator.json` will fail against unupdated operators.

3. **Environment variables control live services.** Renaming a production env var without simultaneously updating the server configuration causes service failure.

4. **SDK version contracts.** If SDK client code is updated to reference new env vars or new routes, the update must be coordinated with the server update.

## Affected QR Payloads

QR codes already in the wild with `BANZAMI:` prefix:

- Cannot be changed (they are printed or cached)
- Must be accepted by the protocol validator during the compatibility window
- New codes generated after the migration must use `BANZA:` prefix
- The compatibility window ends when the team decides to hard-deprecate `BANZAMI:` prefix

**Compatibility window strategy:**

```
Phase 1 (emit + accept both):
  - Generators: emit BANZA: only
  - Validators: accept BANZAMI: and BANZA: both

Phase 2 (accept old, warn):
  - Validators: accept BANZAMI: with deprecation warning in response
  - Log frequency of legacy prefix usage to track when to sunset

Phase 3 (sunset):
  - Validators: reject BANZAMI: with a clear error message pointing to migration guide
  - Timeline: decided after Phase 2 usage data is available
```

## Affected Operator Manifests

Operators currently serve their manifest at `/.well-known/banzami/operator.json`. The migration path:

**For operators (during compatibility window):**

```
Option A â€” Serve both paths:
GET /.well-known/banzami/operator.json  â†’ 200 OK (legacy â€” deprecated)
GET /.well-known/banza/operator.json   â†’ 200 OK (canonical)

Option B â€” Redirect legacy to canonical:
GET /.well-known/banzami/operator.json  â†’ 301 â†’ /.well-known/banza/operator.json
GET /.well-known/banza/operator.json   â†’ 200 OK
```

Option B is preferred â€” it is self-documenting and automatically migrates discovery clients.

**For discovery clients (BanzamIA, conformance tools):**

- Primary: fetch `/.well-known/banza/operator.json`
- Fallback (compatibility window): if 404, try `/.well-known/banzami/operator.json`
- Log when fallback is used â€” this identifies operators that have not yet migrated

## Compatibility Strategy

| Identifier | Strategy | Duration |
|------------|----------|---------|
| `BANZAMI:` QR prefix | Accept during compatibility window; emit `BANZA:` only | Until usage < 1% or explicit sunset decision |
| `BANZAMI-SBX:` QR prefix | Same | Same |
| `/.well-known/banzami/operator.json` | Serve 301 redirect to `/.well-known/banza/operator.json` | Indefinite (redirects are low cost) |
| `BANZAMIA_*` env vars | Document new names; accept legacy names during deprecation period | 1 release cycle |
| `/banzamia` route | 301 redirect to `/banzai` | Indefinite (bookmarks) |
| `/banzamia/*` API routes | 301 redirect or keep serving; deprecation header in responses | 1â€“2 release cycles |

**Principle:** Old names are not canonical. Canonical names are documented as the target. Compatibility is a transition tool, not a permanent state.

## Deprecation Strategy

For each changed identifier, add a deprecation notice:

**QR prefix deprecation notice** (in validator response when `BANZAMI:` detected):

```json
{
  "status": "valid",
  "warnings": [{
    "code": "DEPRECATED_QR_PREFIX",
    "message": "QR prefix BANZAMI: is deprecated. New codes should use BANZA:. See banzami.org/migration/qr-prefix"
  }]
}
```

**Operator manifest redirect** (HTTP response):

```
HTTP/1.1 301 Moved Permanently
Location: /.well-known/banza/operator.json
Deprecation: true
Link: <https://banzami.org/migration/operator-manifest>; rel="deprecation"
```

**Route redirect** (Next.js `next.config.js`):

```js
async redirects() {
  return [
    {
      source: '/banzamia/:path*',
      destination: '/banzai/:path*',
      permanent: true,
    },
  ]
}
```

## Test Requirements

Every change to a wire format identifier requires tests at two levels:

### 1. Unit tests â€” canonical name

```
MUST PASS: validator accepts BANZA: prefix
MUST PASS: generator emits BANZA: prefix (not BANZAMI:)
MUST PASS: discovery client fetches /.well-known/banza/operator.json
```

### 2. Compatibility tests â€” legacy name

```
MUST PASS (during compatibility window): validator accepts BANZAMI: prefix
MUST PASS (during compatibility window): /.well-known/banzami/ returns 301
MUST FAIL (after sunset): validator rejects BANZAMI: prefix with DEPRECATED_QR_PREFIX error
```

### 3. Regression tests

```
MUST PASS: existing operator manifest (legacy path) is still discoverable via redirect
MUST PASS: QR code with legacy prefix is still scannable during compatibility window
```

### 4. Migration tests

```
MUST PASS: operator serving only new path is discovered correctly
MUST PASS: operator serving redirect is discovered correctly
MUST PASS: new QR code with BANZA: prefix is validated correctly
```

3. **SDK update** (Wave 6) â€” documentation and any code changes

**Wire format changes (QR prefixes, operator manifest)** are the highest-coordination items and should be executed in a dedicated sub-wave within Wave 5, after env vars are stable:

